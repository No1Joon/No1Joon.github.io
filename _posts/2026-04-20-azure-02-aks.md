EKS, GKE와 함께 세계 3대 K8s 매니지드 서비스로 꼽히는 **AKS(Azure Kubernetes Service)**는 Azure 생태계와의 강력한 통합(Active Directory, Policy, Monitor 등)을 무기로 엔터프라이즈 환경에서 맹활약 중입니다.

AKS를 프로덕션에 올릴 때 가장 먼저 맞닥뜨리는 고민거리이자 핵심 설계 요소인 **네트워크 모델과 Ingress 구성**을 살펴볼게요.

## AKS의 심장, 두 가지 CNI 모델

쿠버네티스의 노드들이 속한 전통적인 VNet(VPC)과, 그 위에 둥둥 떠 있는 논리적인 컨테이너(Pod)의 네트워크를 어떻게 연결할 것인가? AKS 클러스터 생성 시 아래 두 가지 중 하나를 택해야 합니다. (생성 후 변경 불가!)

| 비교 항목 | Kubenet (기본 네트워크) | Azure CNI (고급 네트워크) |
|---|---|---|
| **Pod IP 할당 방식** | 노드 내부의 논리적 브릿지 범위에서 할당됨 | **VNet의 실제 사설 IP**를 직접 할당받음 |
| **IP 주소 소비량** | 낮음 (Node 개수 정도의 IP만 필요) | **매우 높음** (모든 Pod마다 서브넷 IP를 하나씩 소진) |
| **VNet 밖과의 통신** | 노드 IP로 매핑(NAT)되어 나감 | **Pod IP 그대로 나감** (라우팅 직관적) |
| **추천 환경** | IP 대역이 부족한 레거시 엮인 하이브리드 클라우드 | 대규모 트래픽, 높은 성능, 온프레미스와의 VPN/ExpressRoute 직결 모델 |

"우리는 VNet IP 대역(CIDR)이 `10.0.0.0/16`처럼 널찍해서 초당 생성/소멸하는 수백 개의 Pod에 IP를 펑펑 줘도 부담이 없다!" 라면 성능이 우월한 **Azure CNI**를 택하는 편이 낫습니다. IP 고갈 위험이 있다면 Kubenet이 강제됩니다.

<div class="callout why">
  <div class="callout-title">Azure CNI Overlay의 등장</div>
  기존 Azure CNI의 치명적인 단점인 'IP 고갈 문제'를 해결하기 위해 <strong>Azure CNI Overlay</strong> 모드가 추가되었습니다. 이는 노드에만 VNet IP를 주고 Pod에는 별도의 Overlay IP를 할당하되, Azure CNI 수준의 고성능 트래픽 흐름을 제공합니다. 최근 신규 구축 시 가장 권장되는 모델이에요.
</div>

## 클러스터로 유저를 모시는 방법: Ingress

수십 개의 Pod에 트래픽을 뿌려주는 Ingress Controller로 무엇을 쓸 지도 중요한 결정 사항입니다.

### 선택지 1: 오픈소스 NGINX Ingress
가장 무난하고 널리 쓰이는 패턴이에요. AKS 앞에 붙는 Azure Load Balancer(L4)가 트래픽을 단순히 받아주고, 클러스터 내부의 NGINX 파드가 L7 라우팅(URI 분기 등)을 전담합니다. 

### 선택지 2: Application Gateway Ingress Controller (AGIC)
Azure의 관리형 L7 로드밸런서인 Application Gateway를 쓰는 방법이에요. AGIC라는 부품을 K8s에 얹어주면, K8s의 `Ingress` yaml 파일을 읽어서 Application Gateway의 라우팅 룰을 자동으로 업데이트해 줍니다.
- 장점: 무거운 L7 라우팅 부하와 WAF(웹 방화벽) 연산을 K8s 클러스터 밖(App Gateway)으로 밀어내어 노드 자원을 아낄 수 있습니다. 만약 앞서 다룬 Azure CNI를 쓰고 있다면 파드 IP로 곧장 트래픽이 직행하므로 레이턴시도 환상적입니다.
- 단점: NGINX보다 설정 적용이 몇 초 느리고, App Gateway의 고정 요금이 꽤 큽니다.

## 정리

- AKS 네트워크 설계 시, VNet 통신이 빈번하고 IP가 넉넉하다면 **Azure CNI (또는 Overlay)**를 채택하세요.
- IP가 극도로 부족할 때만 **Kubenet** 모델로 노드 뒤에 파드들을 숨기세요.
- WAF 통합과 빵빵한 파드 직결형 라우팅 성능이 고프다면 **AGIC(Application Gateway)** 모델을 띄우고, 가성비와 유연성을 중시한다면 **NGINX Ingress**를 쓰세요.
- (앞선 개념 복습) App의 Azure 리소스 제어 권한은 반드시 **Managed Identity** (Workload Identity)로 K8s Service Account와 결합하여 전달해야 합니다.

AKS로 모던 앱 뼈대를 세웠습니다. 다음에서는 쿠버네티스까지 갈 필요도 없이 클릭-투-배포를 지원하는 **Azure의 매력적인 PaaS 서비스 라인업(App Service, Functions, Cosmos DB 등)**을 살펴보겠습니다.
